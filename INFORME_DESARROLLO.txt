================================================================================
  INFORME DE DESARROLLO – SISTEMA DE GESTIÓN DE TURNOS VETERINARIA
  Trabajo práctico / recuperatorio POO (Python) – nivel secundario / técnico
================================================================================

Este documento resume el propósito del sistema, cómo está organizado el código,
cómo se usa en ejecución, y relaciona cada tema de la “cheat sheet” de POO con
lo que hicimos (cuándo aplica y cuándo no).


--------------------------------------------------------------------------------
1. QUÉ PROBLEMA MODELO EL SISTEMA
--------------------------------------------------------------------------------

Dominio de problema (recorte de la realidad):
  Una clínica veterinaria donde hay dueños, mascotas, veterinarios, consultorios
  y turnos. Se necesita registrar datos, agendar visitas, evitar conflictos de
  horario y guardar la información entre ejecuciones.

Modelo computable:
  El programa en Python representa ese dominio con clases (plantillas) y objetos
  (instancias con datos concretos), más una capa que aplica reglas de negocio y
  otra que persiste en archivos JSON.


--------------------------------------------------------------------------------
2. ESTRUCTURA DEL PROYECTO (CARPETAS Y ARCHIVOS)
--------------------------------------------------------------------------------

  main.py
    Punto de entrada: crea la clínica, intenta cargar JSON desde data/, abre el
    menú de consola y al salir vuelve a guardar.

  clinica.py  →  clase ClinicaVeterinaria
    Lista de dueños, mascotas, veterinarios, consultorios y turnos. Aquí están
    las validaciones al registrar y al agendar/modificar turnos (solapes, fechas,
    datos obligatorios).

  menu.py  →  clase Menu
    Imprime opciones, lee texto/números del usuario y llama a los métodos de la
    clínica. Algunas acciones están decoradas para guardar en JSON al terminar.

  modelo/
    Persona, Dueno, Veterinario, Animal, Mascota, Consultorio, Turno.
    Representan “cosas del mundo” con atributos y comportamiento.

  dao/
    DuenoDAO, MascotaDAO, VeterinarioDAO, ConsultorioDAO, TurnoDAO, ClinicaDAO.
    Convierten listas de objetos a JSON y viceversa, sin mezclar formato de archivo
    con las reglas de negocio.

  data/
    Archivos duenos.json, mascotas.json, etc. Generados al usar el sistema.

  test/
    Pruebas unitarias (modelo y DAO) y una de integración (carga/guardado global).


--------------------------------------------------------------------------------
3. FLUJO TÍPICO DE EJECUCIÓN
--------------------------------------------------------------------------------

  1) Se ejecuta main.py.
  2) Se instancia ClinicaVeterinaria con un nombre (ej. clínica).
  3) ClinicaDAO.cargar intenta leer data/*.json. Si falta un archivo, se ignora;
     si el JSON es inválido, se muestra error y se puede seguir con listas vacías.
  4) Menu.ejecutar() entra en un bucle: menú principal → submenús (dueños,
     mascotas, veterinarios, consultorios, turnos).
  5) Cada operación que modifica datos y lleva @guardar_automatico llama a
     _guardar() al final, que persiste todas las listas en data/.
  6) Al elegir “Salir” en el menú principal, main vuelve a guardar con el DAO.

Orden de carga importante: primero dueños y mascotas (y veterinarios y
consultorios), después turnos, porque al reconstruir un turno hace falta que ya
existan las entidades enlazadas.


--------------------------------------------------------------------------------
4. FUNCIONAMIENTO POR PARTES DEL SISTEMA
--------------------------------------------------------------------------------

4.1 Registros (dueños, mascotas, veterinarios, consultorios)
  - La clínica valida DNI, teléfono de 11 dígitos, campos no vacíos, duplicados
    (DNI de dueño, matrícula de veterinario, número de consultorio).
  - Las mascotas se asocian a un dueño ya existente; se evita duplicar el mismo
    nombre de mascota para el mismo dueño.

4.2 Turnos
  - Un turno une: mascota, veterinario, fecha/hora y consultorio.
  - Duración fija: 30 minutos (constante en Turno).
  - Solo se permiten minutos de inicio 00, 15, 30 o 45 y segundo 0.
  - Al agendar en “modo normal”, la fecha/hora debe ser futura.
  - No puede haber solape entre turnos ACTIVOS si comparten veterinario, consultorio
    o mascota (se comparan intervalos [inicio, fin)).
  - cancelar_turno marca estado “Cancelado”.
  - modificar_turno vuelve a validar solapes y fecha futura.
  - restaurar_turno se usa al cargar desde JSON: conserva id y estado y no exige
    fecha futura (permite turnos pasados en archivo).

4.3 Eliminaciones
  - No se puede eliminar mascota, veterinario o consultorio si tienen turnos
    activos futuros asociados.
  - No se elimina un dueño si aún tiene mascotas.

4.4 Consultas de reportes
  - Por fecha del día, por matrícula de veterinario, por mascota (nombre + DNI
    dueño), por DNI de dueño. Solo listan turnos en estado “Activo”.

4.5 Persistencia (DAO + JSON)
  - Cada DAO escribe una lista en un archivo con indentación legible.
  - Los turnos guardan nombres y claves necesarias para reconstruir objetos al
    cargar; después de cargar se sincroniza el contador de ids de Turno para que
    los nuevos turnos no repitan ids viejos.


--------------------------------------------------------------------------------
5. PRUEBAS
--------------------------------------------------------------------------------

  Se usa unittest. Hay tests de validación de registro, de turnos (solapes,
  minutos, cancelar, modificar), de consultas y de guardado/cargado de DAO.
  Comando típico:  py -m pytest test/ -q


================================================================================
  RELACIÓN CON LA “CHEAT SHEET” DE POO (TÉRMINO POR TÉRMINO)
================================================================================

Para cada ítem se indica si aplica a ESTE desarrollo, cómo, o por qué no aplica.
(No obliga a que el TP implemente todos los patrones avanzados: muchos son
conceptos generales de la materia.)


--------------------------------------------------------------------------------
A) MODELO COMPUTABLE, DOMINIO, CLASE Y OBJETO
--------------------------------------------------------------------------------

  • Modelo computable / dominio de problema / realidad
    APLICA. El dominio es la clínica veterinaria (turnos, pacientes, personal,
    espacios). El código es el modelo computable ejecutable.

  • Clase
    APLICA. Ej.: Turno, Mascota, ClinicaVeterinaria son plantillas que definen
    atributos y métodos.

  • Objeto (instancia)
    APLICA. Ej.: cada Dueno en la lista _duenos es un objeto con DNI y nombre
    concretos; cada Turno en _turnos tiene su propia fecha y estado.


--------------------------------------------------------------------------------
B) PILARES / PRINCIPIOS CLÁSICOS DE POO
--------------------------------------------------------------------------------

  • Encapsulamiento
    APLICA. Atributos con prefijo “_” (convención de uso interno/protegido) y
    acceso preferente mediante getters (get_dni, get_nombre, etc.). Las reglas
    fuertes (DNI, solapes) están en ClinicaVeterinaria y Turno, no en el menú.

  • Herencia
    APLICA. Persona → Dueno y Veterinario; Animal → Mascota. Se reutiliza lo
    común (datos de persona / animal) y se agrega lo específico (dirección,
    matrícula, dueño de la mascota, etc.).

  • Polimorfismo
    APLICA de forma sencilla. Ej.: todas las Personas entienden mostrar_info(),
    pero Dueño y Veterinario lo implementan distinto. El menú puede llamar
    mostrar_info sobre distintos tipos sin saber el detalle interno.

  • Abstracción
    APLICA. Persona y Animal son clases abstractas (ABC): no se instancian solas;
    definen una interfaz mínima (mostrar_info) y datos comunes. El usuario del
    modelo usa “Persona” o “Animal” sin necesitar saber cada subtipo al escribir
    listas polimórficas.


--------------------------------------------------------------------------------
C) METODOLOGÍAS DE DESARROLLO (CASCADA, INCREMENTAL, ETC.)
--------------------------------------------------------------------------------

  • Cascada / incremental / iterativo / Agile / iterativo-incremental / TDD
    PARCIAL / NO COMO PROCESO FORMAL. El trabajo entrega un producto funcionando
    con pruebas, pero la consigna no exige documentar un ciclo de vida software
    ni reuniones Agile. En la práctica personal pudo haber iteración (probar,
    corregir), pero no está instrumentado como “sprints” ni TDD estricto (tests
    escritos antes que el código en cada paso). Se puede decir en defensa: “usé
    un enfoque incremental informal: primero modelo, luego clínica, menú, DAO y
    tests”.


--------------------------------------------------------------------------------
D) CONVENCIONES DE NOMBRES (PEP 8 / ESTILO)
--------------------------------------------------------------------------------

  • PascalCase para clases, snake_case para métodos y variables
    APLICA en el código Python del proyecto (Turno, registrar_dueno, etc.), en
    línea con la cheat sheet y PEP 8.


--------------------------------------------------------------------------------
E) PATRONES DE DISEÑO AVANZADOS (CHEAT SHEET)
--------------------------------------------------------------------------------

  • Singleton
    NO APLICA de forma explícita. No hay una única instancia global forzada por
    clase (no se bloquea __new__). Hay una instancia de clínica creada en main y
    se pasa al menú: es composición/agregación simple, no el patrón Singleton.

  • Decorator (patrón estructural GoF)
    PARCIAL. El proyecto usa un decorador de Python (@guardar_automatico) que
    envuelve funciones del menú para agregar persistencia. Es el mecanismo de
    decorador del lenguaje; no es el patrón Decorator clásico con clases Wrapper
    encadenadas. En defensa: “decorador funcional para cross-cutting de guardado”.

  • Iterator (patrón)
    NO COMO PATRÓN PROPIO. Se usa el iterador que ya da Python al recorrer listas
    (for t in self._turnos). No hay una clase Iterator custom ni __iter__ propio
    en una colección del dominio.


--------------------------------------------------------------------------------
F) PRINCIPIOS SOLID (RESUMEN DE LA CHEAT SHEET)
--------------------------------------------------------------------------------

  • SRP – Responsabilidad única
    APLICA en lo básico: modelo describe entidades; ClinicaVeterinaria concentra
    reglas de negocio; DAO solo serializa; Menu solo interfaz. Aun así, Clinica
    es grande (muchas responsabilidades relacionadas); en un sistema mayor se
    podría partir en servicios.

  • OCP – Abierto/cerrado
    PARCIAL. Agregar una nueva entidad similar implicaría tocar menú y clínica;
    no hay plugins formales. Para el alcance del TP es aceptable.

  • LSP – Sustitución de Liskov
    APLICA en la medida en que Dueño y Veterinario se usan como Persona donde
    corresponde (herencia sin romper expectativas obvias de mostrar_info / getters).

  • ISP – Segregación de interfaces
    NO FUERTE. No hay interfaces pequeñas en Python aparte de ABC con pocos
    métodos; Persona/Animal ya son interfaces mínimas.

  • DIP – Inversión de dependencias
    PARCIAL. Menu depende de la clase concreta ClinicaVeterinaria, no de una
    abstracción inyectable. El DAO accede a atributos internos de la clínica al
    guardar (acoplamiento práctico para un TP). En un sistema enterprise se
    abstraería un repositorio.


--------------------------------------------------------------------------------
G) COLECCIONES, ITERADORES Y GENERADORES (TEORÍA DE LA CHEAT SHEET)
--------------------------------------------------------------------------------

  • Lista, tupla, conjunto, diccionario, pila, cola (tabla conceptual)
    APLICA lo esencial: el dominio usa principalmente listas (colecciones ordenadas
    de objetos). No se usa un diccionario como estructura principal de datos en
    memoria (las búsquedas son lineales). Tuplas pueden aparecer en uso mínimo
    del lenguaje; no es el foco del diseño.

  • Iteradores (objeto que recorre una colección)
    APLICA de forma implícita con for sobre listas.

  • Generadores (yield)
    NO APLICA. No hay funciones generadoras en el código del TP.


--------------------------------------------------------------------------------
H) MANEJO DE EXCEPCIONES
--------------------------------------------------------------------------------

  APLICA. Se usa raise ValueError / TypeError para datos inválidos; try/except en
  main al cargar JSON y en el menú para capturar errores y mostrar mensaje sin
  cortar el programa de golpe. No se usa en todos los bloques la rama else/finally
  de la teoría, pero el esquema try/except sí refleja el tema de la cheat sheet.


--------------------------------------------------------------------------------
I) PATRÓN DAO
--------------------------------------------------------------------------------

  APLICA claramente. Cada *DAO separa persistencia (JSON, rutas, os.makedirs) de
  la lógica de ClinicaVeterinaria. ClinicaDAO orquesta guardar/cargar todos los
  archivos. La cheat sheet menciona que facilita pruebas: en este proyecto hay
  tests que guardan y cargan usando los DAO.


--------------------------------------------------------------------------------
J) TDD (TEST DRIVEN DEVELOPMENT)
--------------------------------------------------------------------------------

  NO APLICA de manera estricta (no se documenta el ciclo Red-Green-Refactor ni
  que cada feature nació de un test previo). SÍ HAY tests que validan el
  comportamiento, alineado con la consigna de “pruebas simples”.


--------------------------------------------------------------------------------
K) ALINEACIÓN CON LA CONSIGNA DEL TRABAJO PRÁCTICO
--------------------------------------------------------------------------------

  - Clases con atributos y métodos: sí (modelo + clínica + menú + DAO).
  - Herencia Persona → Dueño/Veterinario: sí.
  - Encapsulamiento: sí (atributos con _ y accesos controlados).
  - Validaciones: sí (fechas, solapes, datos obligatorios).
  - Comentarios y PEP 8: el código se ajusta a convenciones razonables (p. ej.
    longitud de línea 88 caracteres con flake8).
  - Menú CRUD y consultas: sí.
  - JSON: sí.
  - Pruebas: sí (pytest/unittest).


--------------------------------------------------------------------------------
CIERRE
--------------------------------------------------------------------------------

El sistema cumple el rol de “aplicación de consola con modelo orientado a objetos,
persistencia y pruebas”. Los temas centrales de la cheat sheet (clase/objeto,
encapsulamiento, herencia, polimorfismo, abstracción, excepciones, DAO, nombres)
están presentes en el código. Los temas más avanzados o de proceso (Singleton
clásico, Iterator patrón propio, generadores, TDD formal, metodologías documentadas)
no son requisito del enunciado y en general no aplican tal cual, o solo de forma
parcial como se explicó arriba.
